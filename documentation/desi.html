Desi

The Future of Standalone Server Orchestration

Overview

The Desi Interface is specifically designed for standalone servers and internal lab machines (like the powerful Desi-Lab servers). It is optimized for rapid iteration, immediate feedback, and bypassing the queue times often associated with Slurm clusters.

No Manual Source Push: SlurmRay automatically scans your local imports and pushes necessary code. You only need to specify data files or non-Python resources in the files argument.

Best Practices for Desi

1. Intelligent Project Naming

Use the same project_name for all calculations belonging to the same project. SlurmRay checks the hash of your requirements: if they haven't changed, the remote virtual environment is reused instantly, avoiding redundant installations.

2. Automatic Dependency Detection

While SlurmRay is smart, it has limits. Static imports are detected perfectly, but dynamic imports (e.g., importlib.import_module) or files accessed via dynamic paths in open() might be missed. If you encounter "ModuleNotFound" errors on the server, add the missing items to the files list manually.

3. Resource Monitoring with CLI

Always use the SlurmRay CLI to monitor your jobs: 

slurmray desi
This allows you to see real-time job status, cancel stalled tasks, and open the Ray Dashboard via an automated SSH tunnel.

4. Automatic Cleanup

To prevent server congestion, all project files and virtual environments are automatically deleted after the specified retention_days (default is 1 day).

Complete Example

Notice how the launcher automatically detects that src.my_model is a local dependency and pushes it. 

from slurmray import RayLauncher
from src.my_model import train

# Configure for Desi Server
launcher = RayLauncher(
    project_name="alpha_model_v1",
    cluster="desi",        # Activates Desi Interface
    files=["data/raw_data.parquet"], # Only data, code is auto-detected!
    num_gpus=1,
    memory=32
)

# Execute Remote Task
result = launcher(train, args={"lr": 0.001})
print(f"âœ… Result: {result}")

Log Locations

If something goes wrong, you can find logs in:

  • Local: logs/RayLauncher.log (detailed bridge logs)
  • Remote: ~/slurmray-server/{project_name}/.slogs/server/ (standard output of your task)

Argument Reference (Desi-Specific)

Please provide precise resource requirements (CPU, RAM, GPU) to help optimize the queue for other users.

Argument Default Description
project_name None Required. Consistent naming allows virtual environment reuse.
cluster "curnagl" Set to "desi" for standalone server execution.
num_cpus 4 Number of CPUs to reserve. Please be precise to optimize scheduling.
memory 20 RAM in GB to reserve. Please be precise to optimize scheduling.
num_gpus 0 Number of GPUs to reserve. Please be precise to optimize scheduling.
files [] Extra data/config files to sync. (Code is auto-detected).
retention_days 1 Days until automatic deletion of project and venv (1-30).
node_nbr 1 Inapplicable: Desi is a single-node server.
modules [] Inapplicable: Desi does not use a module system.
force_reinstall_venv False Forces a clean re-installation of the virtual environment.
force_reinstall_project False Cleans project files on the server before upload.
asynchronous False If True, returns a tracking object immediately.
